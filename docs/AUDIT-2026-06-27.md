## Findings

### [HIGH] [PERF] pdf-proxy cold start dominates Drive PDF open
**Where:** `src/lib/pdfViewerUrl.ts:166-180`, `supabase/functions/pdf-proxy/*`
**Why it matters:** On native, Drive `/view` URLs now route through pdf-proxy → pdf.js viewer. On a cold edge function the first request can be 800–1500 ms before a single byte hits pdf.js, which the UI surfaces as the 63% "stuck" loader the user reported pre-fix. Web stayed on `/preview` so the regression risk only exists on APK.
**Fix:**
- Add a Supabase Scheduled Function ping (every 5 min during 06:00–23:00 IST) to keep the proxy warm. Pattern already in repo: `.github/workflows/supabase-keepalive.yml` — extend it to include `pdf-proxy`.
- Stream the upstream response (`return new Response(upstream.body, …)`) instead of buffering with `arrayBuffer()` — cuts TTFB by ~300 ms on 5–10 MB notes.
- Add a `Cache-Control: public, max-age=86400, immutable` header so the WebView reuses cached PDFs between sessions.

### [HIGH] [RELY] No fallback if pdf-proxy fails on native
**Where:** `src/lib/pdfViewerUrl.ts` native branch
**Why it matters:** If pdf-proxy 5xx's or the edge runtime is cold-failing, the user sees a blank pdf.js shell with no recovery path. We removed the Drive `/preview` fallback for native to fix the blank-screen.
**Fix:** Wrap pdf-proxy resolution in a 6-second timeout; on failure, hand off to `@capacitor/file-viewer` (already in the plugin set per docs) with the original Drive URL. Add a "Try in browser" link in the error state that opens via `openExternal()`.

### [MEDIUM] [OBS] pdf-proxy has no structured logs
**Where:** `supabase/functions/pdf-proxy/index.ts`
**Why it matters:** When a user reports a stuck PDF we can't tell from Supabase logs whether it was upstream 403 (Drive permission), upstream timeout, or proxy cold-start. The Sentry breadcrumb in `[pdf-debug]` lives client-side only.
**Fix:** Emit one structured JSON line per request: `{requestId, upstreamUrl, upstreamStatus, upstreamMs, totalMs, byteSize}`. Cap upstream URL host-only to avoid logging tokens.

### [MEDIUM] [UX] Overlay-sentinel cleanup race in `LessonView` chip PDF
**Where:** `src/pages/LessonView.tsx:874-894`
**Why it matters:** The chip-PDF effect pushes `{pdfFullscreen: true}` and on unmount only flips the flag via `replaceState`. If two PDFs open in quick succession (user taps chip, then taps another chip <300 ms later), the first effect's cleanup can clear the second's sentinel, breaking hardware-back. The dedicated `useOverlayHistorySentinel` hook (which uses unique keys per mount) already exists — this surface is the only remaining hand-rolled one.
**Fix:** Replace the inline `useEffect` with `useOverlayHistorySentinel(!!selectedPdf, () => setSelectedPdf(null))`.

### [MEDIUM] [PERF] `MahimaGhostPlayer` re-pushes `playerFullscreen` on every remount
**Where:** `src/components/video/MahimaGhostPlayer.tsx:155`, `:216`
**Why it matters:** Rapid orientation flips during a network blip can stack multiple sentinels on the history. Each takes one hardware back press to unwind — user reports "back press doing nothing" until they tap 2–3 times.
**Fix:** Guard the push with `if (window.history.state?.playerFullscreen) return;` before `pushState`.

### [MEDIUM] [MAINT] `recordDecision` ring uses `setTimeout(…, 250)` fallback
**Where:** `src/hooks/useAndroidBackButton.ts:109`
**Why it matters:** Older WebViews (Android 7/8) without `requestIdleCallback` end up writing 4 KB to localStorage every back press → main-thread jank visible as a 30–60 ms tap stall. Low impact (back press is already a frame-blocking gesture) but worth noting.
**Fix:** Throttle the persist call to once per second instead of per press; the ring is already capped at 50 entries.

### [MEDIUM] [CONFIG] `webContentsDebuggingEnabled` not guarded by build flavor
**Where:** `capacitor.config.ts`
**Why it matters:** If `webContentsDebuggingEnabled: true` ships in release APK, Play Protect and corporate MDMs will flag it; competitors can attach Chrome DevTools to a production install.
**Fix:** `webContentsDebuggingEnabled: process.env.NODE_ENV !== "production"`.

### [LOW] [PERF] `prefsCache.hydratePrefsCache()` not wired into boot
**Where:** New `src/lib/perf/prefsCache.ts`, not yet called from `src/main.tsx`
**Why it matters:** The utility lands but is unused until a consumer adopts it. Zero risk; opportunity not yet realised.
**Fix:** Add `void hydratePrefsCache();` immediately after `createRoot(...).render(...)` in `src/main.tsx`. Then migrate `nb:theme`, `nb:last-route`, `nb:current-batch` reads to `getPref()`.

### [LOW] [A11Y] Breadcrumb back arrow has no `title`
**Where:** `src/components/course/Breadcrumbs.tsx:104`
**Why it matters:** Screen readers announce only "Back" via aria-label; a `title` would also give long-press tooltip on desktop and TalkBack hint.
**Fix:** Add `title="Back to previous level"`.

### [LOW] [MAINT] `useAndroidBackButton.ts` ring localStorage key is unscoped
**Where:** `src/hooks/useAndroidBackButton.ts:64`
**Why it matters:** `LS_KEY = "nb:back-decisions"` collides if two apps share an origin (won't happen in APK; could happen in dev preview iframes).
**Fix:** Suffix with `appId` from `capacitor.config.ts`.

## Wins (what's done right)

- Back-button chain is genuinely well thought through: overlay sentinel → auth-route → exit-window short-circuit → real trail → parent-map → history fallback. The fact that `EXIT_ROUTES` short-circuits BEFORE the trail prevents the "trapped on dashboard forever" failure mode that most apps ship.
- `Button` component bakes haptic + `active:scale-[0.97]` into every variant with sane defaults per variant (light/medium/no-haptic). The soft-touch pass is effectively complete app-wide.
- `nativeStorage.supabaseAuthStorage` correctly routes auth tokens to Keystore/Keychain via Preferences, with idempotent migration off localStorage. No tokens at rest in WebView storage.
- `lazyWithRetry` + `LazyPdfViewer` + dynamic `loadAppPlugin` keep the initial chunk small; haptics, filesystem, app, preferences are all lazy.
- `BackButtonDebug` page with the ring buffer + persisted decisions is genuinely production-grade observability.
- `useOverlayHistorySentinel` uses per-mount unique keys — race-safe by design.
- Route-aware parent map (`PREFIX_PARENT_RULES`, `resolveFromParam` with sessionStorage fallback) survives refresh + popstate edge cases.

## Open Questions

- Should pdf-proxy enforce a Drive-host allowlist (drive.google.com, docs.google.com only) to harden against SSRF? Current behaviour proxies arbitrary user-supplied URLs.
- Do we want a one-time client-side cache for proxied PDFs in IndexedDB (per file_id, 50 MB cap) so repeat opens are instant offline? Pairs naturally with the existing `useDownloads` flow.
- Is there appetite to drop Android 7 (SDK 24) support? `requestIdleCallback`, `OffscreenCanvas`, and modern WebView features would all become safely assumable.
