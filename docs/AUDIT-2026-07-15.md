## Findings

### [HIGH] [SEC] [#18 PII enumeration] Authenticated users can read every profile's email + mobile
**Where:** `supabase/migrations/20260221042958_…sql:16` — policy `"Public profiles are viewable by everyone" ON public.profiles FOR SELECT USING (true)` never had `TO anon` narrowed **or** a column filter added. `20260301071250` only blocked `anon`. `20260710110714` `REVOKE SELECT … FROM anon` closes anon but leaves the `authenticated` grant + `USING (true)` policy intact.
**Symptom:** Any signed-in user can run `supabase.from('profiles').select('id,full_name,email,mobile').range(0,999)` and page through every user's contact info.
**Root:** The public-read policy targets the whole row for `authenticated`. The `profiles_public` view (`id, full_name, avatar_url`) exists but isn't the only client-reachable path — the base table is still readable.
**PoC (described, not executed against prod):**
```bash
curl -sS "$VITE_SUPABASE_URL/rest/v1/profiles?select=id,full_name,email,mobile&limit=5" \
  -H "apikey: $ANON" -H "Authorization: Bearer $USER_JWT"
# Returns rows for users other than $USER_JWT.sub
```
**Fix (proposed migration, not applied):**
```sql
-- Restrict base-table SELECT to owner + admin only; keep public-fanout via profiles_public view.
DROP POLICY IF EXISTS "Public profiles are viewable by everyone" ON public.profiles;
CREATE POLICY "Users read own profile" ON public.profiles
  FOR SELECT TO authenticated
  USING (auth.uid() = id OR public.has_role(auth.uid(), 'admin'::app_role));
-- profiles_public view already exposes safe columns (id, full_name, avatar_url) to authenticated.
GRANT SELECT ON public.profiles_public TO authenticated;
```
Then audit any client query that reads `profiles.email/mobile` and repoint to `profiles_public` or an authenticated RPC.
**Regression guard:** Playwright `e2e/receipts-rls.spec.ts`-style spec: `expect(rows.length).toBe(1)` after `select('email').range(0,999)` as a non-admin user.

### [MEDIUM] [SEC] [#8 XSS surface] CSP retains `'unsafe-inline'` + `'unsafe-eval'` on `script-src`
**Where:** `index.html` meta CSP (documented in `docs/briefs/08-security.md §3`).
**Symptom:** Any injected `<script>` payload (community post body, chatbot echo, admin CMS blob) executes and immediately calls `supabase.auth.getSession()` for a full account-takeover token.
**Root:** Boot inline script + Razorpay checkout + pdf.js worker eval force the loose directives.
**Fix (later, not applied):** externalize boot to `/boot.js` with a SHA-256 hash directive; upgrade pdf.js to a build with `disableEval=true`; move Razorpay to their nonce-supporting integration. Until then, DOMPurify every user-authored HTML sink (already covered by `src/test/dompurify-xss.test.ts` — keep it green).
**Regression guard:** run `dompurify-xss.test.ts` in CI (already present in Vitest suite).

### [MEDIUM] [SEC] [#10 SSRF] `pdf-proxy?kind=url&url=…` accepts arbitrary URLs from any signed-in user
**Where:** `supabase/functions/pdf-proxy` (called via `remotePdfProxyUrl` in `src/lib/pdfViewerUrl.ts:199`).
**Symptom:** A signed-in user can craft `?token=<own JWT>&kind=url&url=http://169.254.169.254/latest/meta-data` and turn the edge function into an SSRF probe against Supabase's cloud metadata / internal hosts.
**Root:** `renderablePdfUrl()` only whitelists `isJsDelivrCdn|isGithubRawCdn|isAzureBlobCdn|isGithubStoragesCdn|isNaveenBharatStorage` for **client-side routing**, but `remotePdfProxyUrl()` itself is called from those paths and the edge function accepts any URL string without re-validating server-side.
**Fix (proposed, not applied):** in `supabase/functions/pdf-proxy/index.ts`, hard-enforce the same host allow-list server-side. Reject anything else with `400`. Explicitly deny RFC1918 + link-local + `localhost` after DNS resolution.
**Regression guard:** unit test in the edge function that asserts `pdf-proxy?kind=url&url=http://169.254.169.254/` returns `400`.

### [MEDIUM] [SEC] [#6 storage listing] Public buckets from 2026-03 migrations
**Where:** `supabase/migrations/20260301050119…sql:8` and `20260301064913…sql:6` create buckets with `public = true`.
**Symptom:** Anonymous browsing of `${SUPABASE_URL}/storage/v1/object/public/<bucket>/…` lists any object whose name an attacker can guess. Object-level RLS was tightened in `20260715140648` for chat-attachments/receipts/comment-images, but the bucket-level `public` flag still bypasses SELECT policies for anonymous public URLs.
**Root:** `public=true` at bucket creation was never flipped back.
**Fix (proposed):**
```sql
UPDATE storage.buckets SET public = false WHERE id IN ('chat-attachments','receipts','comment-images');
-- If any UI relies on public URLs, switch it to signed URLs via createSignedUrl().
```
Verify which client code calls `.getPublicUrl(...)` for these buckets before flipping.
**Regression guard:** integration test — anon `fetch('/storage/v1/object/public/receipts/anything.pdf')` must 404.
